Enterprise AI Assistant – Interview Q&A (SQLAlchemy Foundation)

1.  What is an ORM, and why do we use it? An ORM (Object Relational
    Mapper) lets us work with Python objects instead of writing SQL for
    every database operation. SQLAlchemy translates those Python objects
    into SQL queries, making the code cleaner, easier to maintain, and
    database-independent.

2.  Why don’t we use RegisterRequest as the database model?
    RegisterRequest is responsible for validating incoming API data. A
    User entity represents how data is stored in the database. Keeping
    them separate follows the Single Responsibility Principle and
    Separation of Concerns.

3.  Why do ORM models inherit from Base? Base provides a common registry
    for all ORM models. SQLAlchemy uses it to collect metadata about
    tables, columns, and relationships so it knows how to map Python
    classes to database tables.

4.  Why should there be only one Base? Having one Base creates a single
    registry for all entities. If multiple Base objects exist,
    SQLAlchemy treats them as separate registries, making model
    management more difficult.

5.  What is metadata in SQLAlchemy? Metadata is information about the
    database schema, such as table names, columns, primary keys,
    indexes, and constraints. SQLAlchemy uses it to understand how the
    database should look.

6.  What is the Engine? The Engine is the application’s connection to
    the database. It manages database connections, the connection pool,
    and executes SQL generated by SQLAlchemy. Since it is thread-safe,
    it is created only once.

7.  What is a Session? A Session represents one unit of work. It tracks
    changes made to ORM objects and writes them to the database only
    when commit() is called.

8.  Does creating a User object insert a row into PostgreSQL?
    No. Creating a User object only creates it in Python memory. Even
    after session.add(), the object is only tracked by the Session. It
    is stored in the database only after session.commit().

9.  Why is SessionLocal called a factory? SessionLocal is not a Session
    itself. It is a factory that creates a new Session whenever the
    application needs one.

10. Why does every request need its own Session? A Session represents a
    single transaction and is not thread-safe. Giving each request its
    own Session ensures transaction isolation and prevents requests from
    interfering with each other.

11. Why do we call session.commit()? commit() persists all tracked
    changes to the database as a single transaction. This ensures either
    all changes are saved or none of them are if an error occurs.

12. What happens if an exception occurs before commit()? The changes
    remain only in memory and are never written to the database because
    the transaction was not committed.

13. Why do we use yield instead of return in get_db()? return ends the
    function immediately. yield pauses the function, gives the Session
    to FastAPI, and allows the function to resume later so cleanup code
    can run.

14. Why is db.close() placed inside finally? The finally block executes
    whether the request succeeds or fails. This guarantees that the
    Session is always closed and prevents connection leaks.

15. Who owns the Session lifecycle? FastAPI owns the Session lifecycle.
    It creates the Session, injects it into the request through
    dependency injection, and closes it when the request finishes.

16. What design principle does get_db() follow? The component that
    acquires a resource is responsible for releasing it. Since get_db()
    creates the Session, it is also responsible for closing it.

17. What is the biggest lesson from these topics? Good backend
    development is not about memorizing framework APIs. It is about
    understanding the engineering principles behind them—Single
    Responsibility, Separation of Concerns, Dependency Injection,
    Transaction Management, and proper Resource Management.
